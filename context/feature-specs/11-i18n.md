# Feature 11 — Internationalization (English & Thai)

Read `context/CONTEXT.md`, `context/project-overview.md`, `context/ui-context.md`, and `context/code-standards.md`. Use `vue-best-practices` and `frontend-ui-engineering` skills when implementing the Vue layer.

**Goal:** Let customers and admins use the SPA in **English (`en`)** or **Thai (`th`)**. UI copy, validation messages, toasts, and confirmation emails follow the active locale. Catalog content (movie titles, cinema names) stays as stored in MongoDB — not machine-translated.

**Depends on:** Features 01–10 (full MVP UI and email worker exist).

## Objective

Bangkok-focused cinemas serve Thai- and English-speaking users. Today all UI strings and email templates are hardcoded English. This feature adds a first-class locale switcher, structured translation files, locale-aware formatting, and localized confirmation emails.

**Success looks like:** User switches EN ↔ TH in the header → all customer and admin chrome updates immediately; dates and currency display correctly per locale; confirmation email subject/body match locale at booking time; preference survives refresh; no mixed-language pages after switch.

## Assumptions

1. **Two locales only** for MVP: `en`, `th`. No URL prefix (`/en/...`); locale is client preference + optional cookie for API/email.
2. **Default locale:** Browser `navigator.language` — if it starts with `th`, use `th`; otherwise `en`. User override wins via switcher.
3. **Catalog content is not i18n:** Movie titles, synopses, cinema names from admin/seed remain single-language (often Thai). Only **UI chrome** is translated.
4. **API error bodies stay English** for logs/support; clients map stable **`code`** fields (e.g. `SEATS_UNAVAILABLE`) to translated messages. Do not add server-side message catalogs in MVP.
5. **Gregorian calendar** for showtimes in both locales (`Intl.DateTimeFormat` with `timeZone: cinema.timezone`).
6. **Currency:** THB in both locales; format with `Intl.NumberFormat` (`th-TH` vs `en-US` grouping differs).
7. **Admin UI is translated too** — same switcher in admin shell; admins may operate in either language.
8. **Email locale** is captured at **confirm** time from the user's active SPA locale (`X-Locale` header), **persisted on the `bookings` document**, and used by the email worker — not re-read from client at send time.

## Resolved Decisions

1. **Default locale:** **Browser detect** — `navigator.languages` / `navigator.language`; if any entry starts with `th`, use `th`; otherwise `en`. Saved `localStorage` override takes precedence on return visits.
2. **Booking locale:** **Store `locale` (`en` | `th`) on the `bookings` document** at confirm for support lookup and email rendering (see spec 04 amendment).
3. **Thai typography:** **Noto Sans Thai + Inter** stack via Google Fonts — Thai glyphs render correctly; Latin UI unchanged (see `ui-context.md`).

## Translation Key Conventions

- **Nested namespaces:** `nav.home`, `auth.login.title`, `booking.hold.countdown`, `admin.dashboard.title`, `errors.seats_unavailable`.
- **Match API error codes:** `errors.${snake_case_code}` maps from JSON `code` (e.g. `SEATS_UNAVAILABLE` → `errors.seats_unavailable`).
- **Interpolation:** `{name}`, `{count}`, `{minutes}` — use vue-i18n pluralization for seat counts where needed.
- **No string concatenation** in components; full phrases in JSON for Thai word order.

## Vue Integration

### Browser locale detection (`i18n/index.ts`)

Priority order for `resolveInitialLocale()`:

1. `localStorage.getItem('tbs.locale')` if `en` or `th`
2. First matching entry in `navigator.languages` (fallback `navigator.language`) where tag starts with `th` → `th`, else `en`
3. Fallback `en`

```ts
const SUPPORTED = ['en', 'th'] as const
export type AppLocale = (typeof SUPPORTED)[number]

export function detectBrowserLocale(): AppLocale {
  const tags = navigator.languages?.length
    ? [...navigator.languages]
    : [navigator.language]
  for (const tag of tags) {
    if (tag.toLowerCase().startsWith('th')) return 'th'
  }
  return 'en'
}

export function resolveInitialLocale(): AppLocale {
  const saved = localStorage.getItem('tbs.locale')
  if (saved === 'en' || saved === 'th') return saved
  return detectBrowserLocale()
}
```

### Bootstrap (`main.ts`)

```ts
import { createI18n } from 'vue-i18n'
import en from '@/locales/en.json'
import th from '@/locales/th.json'
import { resolveInitialLocale } from '@/i18n'

export const i18n = createI18n({
```

### Locale switcher (`ui-context.md`)

- Place in **customer header** (home, movie detail, booking flow) and **admin sidebar footer**.
- Control: compact toggle **EN | ไทย** or segmented button; `aria-label` from `t('locale.switcher')`.

### API client error mapping

Centralize in `app/src/api/errors.ts`:

```ts
import { i18n } from '@/i18n'

export function translateApiError(code: string, fallback?: string): string {
  const key = `errors.${code.toLowerCase()}`
  if (i18n.global.te(key)) return i18n.global.t(key)
  return fallback ?? i18n.global.t('errors.generic')
}
```

Use in catch blocks instead of hardcoded strings like `'Could not load movies.'`.

## API & Email Changes

### Confirm booking locale

| Item | Detail |
| ---- | ------ |
| Client | Send `X-Locale: en` or `X-Locale: th` on `POST /api/bookings/confirm` |
| Server | Read header; validate `en` \| `th`; default `en` if missing/invalid |
| Storage | **`locale` field on `bookings` document** — set at confirm, returned in booking DTOs for admin/support |
| Worker | Read `booking.Locale` from MongoDB when rendering email; `renderConfirmation(locale, data)` selects EN/TH template |

### Booking model (spec 04 amendment)

Add to `Booking`:

```go
Locale string `bson:"locale" json:"locale"` // "en" | "th" — UI language at confirm
```

- Required on new bookings after feature ships; existing bookings without `locale` → worker defaults to `en`.
- Admin booking search/detail may display locale badge for support.

### Email templates (spec 09 extension)

| Locale | Subject example |
| ------ | ----------------- |
| `en` | `Your tickets — {movieTitle}` |
| `th` | `ตั๋วของคุณ — {movieTitle}` |

- Separate template funcs or `text/template` files per locale.
- Movie/cinema names in body remain DB language; surrounding copy is translated.
- `email_logs` unchanged; optional `locale` field for debugging (ask first if schema change).

### Out of scope for API i18n

- Translating handler log messages or JSON `message` fields.
- `Accept-Language` negotiation on every public GET (catalog stays locale-agnostic).
- Localized admin audit log action labels (English OK for MVP).

## Code Style

- **Templates:** Prefer `{{ t('key') }}` over `$t` in script; use `$t` only in template if cleaner.
- **Script constants:** Move user-visible strings out of `computed`/`ref` defaults into locale files.
- **Dates:** Replace bare `toLocaleString()` with composable that reads active locale:

```ts
export function useLocaleFormat() {
  const { locale } = useI18n()
  function formatDateTime(iso: string, timeZone: string) {
    return new Intl.DateTimeFormat(locale.value === 'th' ? 'th-TH' : 'en-US', {
      dateStyle: 'medium',
      timeStyle: 'short',
      timeZone,
    }).format(new Date(iso))
  }
  return { formatDateTime, formatTHB: (n: number) => /* Intl.NumberFormat */ }
}
```

- **Go email:** No external i18n library; static templates per locale in Go files.

## Boundaries

- **Always:** Add new user-visible strings to **both** `en.json` and `th.json` in the same PR.
- **Always:** Set `document.documentElement.lang` when locale changes.
- **Always:** Use API `code` for error translation keys, not English `message` text.
- **Ask first:** Adding `locale` to `users` collection; third locale; vue-i18n lazy-loaded async messages.
- **Never:** Machine-translate movie/catalog content in the UI layer.
- **Never:** Mix hardcoded English with `$t` on the same screen after migration.
- **Never:** Break type-check — use typed message keys (`vue-i18n` strict types optional follow-up).
